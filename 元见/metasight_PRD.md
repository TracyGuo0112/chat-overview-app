# MetaSight（元见）产品需求文档（PRD）

## 1. 概述
MetaSight 是面向 18–35 岁用户的数字命理顾问 SaaS，融合传统命理（八字、六爻等）与 AI 对话能力，提供即时咨询、个人命盘管理、人生星图报告、订阅与邀请激励等功能。

本 PRD 基于代码扫描与模块分析结果整理，覆盖当前实现的核心产品能力、关键流程、接口与数据结构、约束假设及待确认事项。

---

## 2. 目标与背景
### 2.1 背景
用户在情感、职业、人生方向决策中，存在“低门槛、即时性、隐私友好”的数字化指引需求。传统线下咨询成本高、预约长；线上泛内容又常缺乏个性化深度。

### 2.2 产品目标
1. 提供可持续的 AI 命理咨询体验（对话 + 工具化推理）。
2. 建立个人命理资产沉淀（八字档案、聊天线程、星图报告）。
3. 通过订阅与配额实现商业化，并保持体验公平性。
4. 通过博客内容 + 多语言支持实现自然流量增长。

### 2.3 成功指标（建议）
- 对话首响成功率、流式完成率
- 人均会话轮次、7日留存
- 星图报告生成成功率、章节重试成功率
- 免费转付费率、订阅续费率
- 邀请码使用率、邀请带来新增占比

---

## 3. 用户角色
1. 访客（未登录）
   - 浏览落地页与博客，了解产品价值。
2. 注册用户（免费）
   - 使用基础咨询能力，管理个人资料与部分命理档案。
3. 订阅用户（付费层级）
   - 获得更高配额/更完整功能（如更多咨询额度、更多档案能力等）。
4. 运营/增长角色
   - 关注内容分发、邀请裂变、基础后台管理。
5. 开发/运维角色
   - 通过 Cron、E2E、监控保障稳定性与可用性。

---

## 4. 需求清单

### 4.1 认证与账号
- 支持登录/注册、邮箱验证、Magic Link/密码相关流程。
- 支持忘记密码、重置密码、修改/设置密码。
- 受保护路由需鉴权，未登录跳转登录页。

### 4.2 AI 对话咨询
- 支持创建/续写会话线程，保存历史消息。
- 支持消息编辑后截断重生成（保证上下文一致）。
- 支持流式输出与断流恢复（resumable stream）。
- 支持追问建议（followup）与输入自动补全（autocomplete）。
- 支持安全拦截（Guard）与模型 fallback。

### 4.3 命理业务能力（八字/六爻/星图）
- 八字档案管理：创建、编辑、归档、删除、置顶、复制。
- 标签体系与数量限制校验。
- 星图报告：启动生成、章节生成、失败章节重试、进度查询。
- 工具化能力：六爻起卦、命理分析辅助工具。

### 4.4 订阅、权益与配额
- 订阅状态同步（写入/更新/过期处理）。
- 权益判定：tier、feature、quota、credit。
- 对话等高频能力需在调用前执行配额检查。
- 合法成功响应后才消耗积分（异常/降级路径不应误扣）。

### 4.5 邀请裂变
- 用户可生成/分享邀请码。
- 新用户可校验并绑定邀请码。
- 满足规则后发放奖励积分，并可查询统计。

### 4.6 内容与多语言
- 博客内容基于 MDX 管理。
- 支持至少 zh/en/zh-Hant 多语言文案。
- 博客支持按 locale/slug/category 查询展示。

### 4.7 质量与稳定性
- 关键业务链路具备 E2E 覆盖。
- 提供错误处理与监控入口（instrumentation）。
- 提供订阅过期的定时任务回收机制。

## 5. 界面/交互逻辑（文本树）

```text
MetaSight Web（按 locale）
├── Landing
│   ├── 首页价值说明
│   ├── 法务页（隐私/条款/Cookies）
│   └── CTA：注册/开始咨询
├── Auth
│   ├── 登录
│   ├── 注册
│   ├── 邮箱验证 / Magic Link 验证
│   └── 重置密码
├── Blog
│   ├── 文章列表（按语言/分类）
│   └── 文章详情（slug）
└── Protected Dashboard
    ├── Chat
    │   ├── 会话列表
    │   ├── 会话详情（流式响应）
    │   ├── 输入区（模式切换、自动补全）
    │   └── 消息工具结果展示（八字/六爻/检索等）
    ├── Bazi
    │   ├── 档案列表
    │   ├── 创建/编辑弹窗
    │   └── 档案详情（图表/运势等）
    ├── Star Map
    │   ├── 生成入口
    │   ├── 进度轮询
    │   └── 报告查看/章节重试
    └── Settings
        ├── 账户资料
        ├── 出生信息
        ├── 订阅信息
        └── 邀请与推荐
```

### 关键交互逻辑（摘要）
1. 聊天发送：输入 -> API 鉴权 -> 安全检查 -> 配额检查 -> 流式响应 -> 落库 -> 成功后扣减积分。
2. 首次会话：线程自动创建并异步生成标题。
3. 编辑重发：截断目标消息后的内容，再按新输入生成后续。
4. 星图报告：创建任务 -> 后台分章节生成 -> 前端轮询进度 -> 支持单章重试。

## 6. API 及关键数据结构说明

### 6.1 关键 API（已识别）
1. `POST /api/chat`
   - 输入：`messages[]`, `threadId`, `mode?`, `truncateFromId?`, `locale?`
   - 输出：流式 UIMessage（SSE）或错误（如 quota_exceeded）
   - 特性：Guard 检测、历史裁剪、模式选择、配额校验、fallback、可恢复流。

2. `POST /api/chat/followup`
   - 输入：最近消息数组
   - 输出：1组追问建议

3. `POST /api/chat/autocomplete`
   - 输入：`text`
   - 输出：补全文本建议

4. `GET /api/chat/{id}/stream`
   - 输入：线程与流上下文
   - 输出：恢复中的流式事件

5. `GET/POST /api/auth/[...all]`
   - 输入：认证协议请求
   - 输出：会话/回调/认证状态

6. `GET /api/cron/expire-subscriptions`
   - 输入：Cron 鉴权请求
   - 输出：过期订阅处理结果（状态更新与额度重置）

### 6.2 Server Actions（核心）
- Bazi：`createBaziAction`、`deleteBaziAction`、`archiveBaziAction` 等
- Star Map：启动报告、章节生成、重生成等
- Auth：忘记密码、重置密码、修改密码等
- Referral：邀请码校验等

### 6.3 关键数据结构（领域）
1. 用户与会话
   - `users`, `sessions`, `accounts`, `verifications`

2. 用户画像与订阅
   - `user_profiles`：性别、出生时区、profileInfo 等
   - `subscription`（creemSubscriptions）：订阅状态、周期、referenceId 等

3. 对话域
   - `chat_threads`：线程元数据（标题、状态）
   - `chat_messages`：消息内容、角色、流状态、反馈等

4. 命理域
   - `bazi_records`：姓名、出生信息、标签、主档标记等
   - `star_map` / `star_map_reports`（从服务行为推断）：配置、章节结果、生成状态

5. 权益域
   - `credits` / quota 计算（按 tier 配置）
   - feature access 与 plan 对应关系

> 注：字段级完整定义以 `lib/server/db/schema.ts` 与 `drizzle/*.sql` 为准。

## 7. 限制与假设
1. 模型供应商与路由策略
   - 假设通过环境变量配置 provider；代码层提供抽象与 fallback，但实际质量受模型与配额影响。

2. 订阅事件一致性
   - 假设支付/认证插件事件可稳定到达并幂等写入；若事件丢失，权益可能短暂不一致。

3. 星图报告耗时
   - 假设采用异步章节生成，前端轮询可接受；极端高峰下可能延迟上升。

4. 配额策略
   - 假设各 tier 的 chat/bazi 等上限由统一配置维护，调用方必须走 entitlement 检查。

5. 多语言一致性
   - 假设文案 key 在各 locale 完整覆盖；缺失项可能回退或显示异常。

6. 当前文档范围
   - 本 PRD 基于代码现状与静态推断，不包含未提交分支与外部平台控制台配置明细。

## 8. 风险与待 Clarify 问题

### 8.1 风险
1. AI 质量与成本风险
   - 高并发与复杂问题下，主模型失败率/延迟抖动可能增加，fallback 频率上升导致成本或体验波动。

2. 订阅与权益不一致风险
   - 支付事件延迟、重复或失败时，可能出现“已支付但未解锁”或“过期未回收”短暂状态。

3. 流式链路复杂性风险
   - 可恢复流 + 消息持久化 + 扣费规则并行，若状态机边界处理不足，易出现重复写入或扣费争议。

4. 命理解释合规/安全风险
   - 需持续强化 Guard 与输出规范，避免高风险场景中的不当建议。

5. 测试覆盖不足风险
   - 若 E2E 未覆盖付费、过期、重试、异常中断路径，线上回归风险较高。

### 8.2 待 Clarify
1. 订阅层级与每档配额的正式产品定义（当前可从配置推断，但需产品确认口径）。
2. Star Map 报告的章节清单、每章 SLA、失败重试上限与用户可见策略。
3. 邀请奖励发放规则（触发时机、上限、反作弊策略）是否与当前实现一致。
4. Guard 拦截策略与人工申诉机制是否需要产品化页面支持。
5. 后台 Admin 能力的对外范围（仅内部可见还是后续产品化）。
6. 博客内容的发布流程（审核、定时发布、SEO 指标归因）是否已有运营 SOP。

## 9. 模块到业务能力映射（摘要）
- Chat 核心：`app/api/chat` + `lib/server/ai` + `lib/server/service/chat`

- 权益系统：`lib/server/service/entitlement` + `lib/server/service/subscription`
- 账号体系：`app/api/auth` + `lib/server/service/auth`
- 增长体系：`lib/server/service/referral` + `content/blog` + `lib/blog`

## 10. 附：典型端到端流程（文本）
1. 新用户首访到咨询
   - 落地页 -> 注册/登录 -> 进入 Dashboard Chat -> 首次提问 -> 线程创建 + 流式回复 -> 会话保存。

2. 免费用户升级订阅
   - 使用中触达配额上限 -> 引导订阅 -> 订阅状态同步 -> entitlement 更新 -> 继续使用高级能力。

3. 命理深度报告
   - 维护出生/档案信息 -> 发起 Star Map 生成 -> 轮询进度 -> 查看章节 -> 失败章节可重试。
